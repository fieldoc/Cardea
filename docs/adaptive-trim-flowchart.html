🎯 What Problem Does Adaptive Trim Solve?

Heart rate doesn't respond instantly to pace changes — there's a physiological lag (typically 15–40 seconds). If you speed up, your HR keeps climbing for a while before it levels off. If you slow down, HR stays elevated before dropping.

Without prediction, the app can only tell you "you're above zone" after you've already been there. With adaptive trim, it projects where your HR is heading and warns you before you leave the zone.

⏱ Per-Tick Projection Formula

Every ~1 second, evaluateTick() computes a projected HR. Here's the formula broken down:

projectedHr = currentHr
+ (hrSlope × horizonSec / 60) ← where HR is trending right now
+ (shortTermTrim × 0.35) ← recent prediction error correction
+ longTermTrim ← your body's systematic bias learned over sessions
+ paceTrendBias ← what HR this pace historically produces

Each component serves a different time scale — from sub-second HR trend to multi-session body knowledge.

① HR Slope (real-time trend)

Tracks the instantaneous rate of HR change in bpm per minute.

instSlope = (currentHr − lastHr) / deltaMinutes

hrSlope = prevSlope × 0.75 + instSlope × 0.25

This is an exponential moving average (EMA). The 75/25 split smooths out beat-to-beat noise while still responding quickly to real trends.

📄 AdaptivePaceController.kt → updateHrSlope()

② Short-Term Trim (error correction)

A self-correcting signal: compares last tick's prediction to the actual HR.

predictionError = actualHr − lastProjectedHr

shortTermTrim = prevTrim × 0.86 + error × 0.14

Decays toward zero (0.86 decay) so it doesn't accumulate forever. Clamped to ±20 bpm. Applied to projection at 35% weight.

Intuition: If the last few predictions were too low (HR came in higher than expected), short-term trim nudges future predictions up.

📄 AdaptivePaceController.kt → evaluateTick()

③ Long-Term Trim (session learning)

Your body's systematic HR bias — are you consistently running hotter or cooler than target?

sessionError = session avg HR − session avg target

longTermTrim = prevTrim × 0.92 + sessionError × 0.08

Updated only at workout end (not per-tick). Persists to disk. After ~12 sessions, this converges on your personal bias. Clamped ±20 bpm.

📄 AdaptivePaceController.kt → finishSession()

④ Pace-HR Bias (historical lookup)

At your current pace, does your HR typically run above or below target?

bucketKey = round(paceMinPerKm × 4) ← 15s/km granularity

avgBucketHr = avg of [bucket-1, bucket, bucket+1]

paceBias = (avgBucketHr − targetHr) × 0.2

Looks up 3 neighboring buckets (±15s/km) for smoothness. Clamped to ±6 bpm. Uses the cross-session paceHrBuckets map.

Example: If at 6:00 min/km your HR averages 155 but target is 145, paceBias = (155-145) × 0.2 = +2 bpm, nudging the projection up.

📄 AdaptivePaceController.kt → lookupPaceBias()

🔭 Projection Horizon & Confidence

How far ahead does it project?

horizon = responseLag × 0.4
clamped to 10–15 seconds

The projection looks ahead by ~40% of your measured response lag. If your body takes 30s to stabilize after a pace change, it projects 12s ahead.

When does the projection activate?

Three confidence tiers based on data accumulated:

  • LOW — <2 sessions AND <80 samples → projections hidden, shows "Learning your patterns"
  • MEDIUM — 2–3 sessions OR 80–179 samples → shows projections with hedged language ("may drift up")
  • HIGH — 4+ sessions OR 180+ samples → full confidence ("drifting up")

⏳ How Response Lag Is Measured

trackSettling() runs every tick and observes zone transitions:

  • When HR leaves the zone (ABOVE or BELOW), a timer starts
  • When HR returns IN_ZONE, the timer stops → that's one "settling sample"
  • Settle-down samples (above→in) and settle-up samples (below→in) are tracked separately
  • At session end, they're averaged and blended with the existing lag estimate

Why it matters: Response lag controls the projection horizon. A runner with a 40s lag needs to be warned earlier than one with a 15s lag.

🪣 Pace-HR Bucket Merging

Buckets are keyed by round(pace × 4) — so bucket 24 = 6:00 min/km, bucket 25 = 6:15, etc.

During session:

Each GPS update accumulates HR samples into the matching bucket:

val bucket = (smoothedPace * 4f).roundToInt()
running.hrSum += hr
running.sampleCount += 1

At session end (merge):

Session averages are weighted-averaged with baseline buckets:

val avg = (baseline.avgHr * baseline.count
         + session.avgHr * session.count)
         / (baseline.count + session.count)
// sample count capped at 10,000

💾 Data Storage: AdaptiveProfile

The entire adaptive learning state is one JSON blob in Android SharedPreferences.

Storage mechanism

  • File: hr_coach_adaptive_profile.xml in app's private SharedPreferences dir
  • Key: "adaptive_profile_json"
  • Format: Gson-serialized JSON of AdaptiveProfile data class
  • Scope: Singleton repository (@Singleton, Hilt-injected)
  • Thread safety: @Synchronized on read/write

📄 AdaptiveProfileRepository.kt

Why SharedPreferences (not Room)?

  • It's a single, small document — not relational data
  • Read on startup, written once per workout end — no need for queries or Flow observation
  • Gson handles the nested Map<Int, PaceHrBucket> naturally
  • No migration headaches — the default-valued data class handles missing fields gracefully

📦 AdaptiveProfile Fields

Defined in domain/model/AdaptiveProfile.kt:

data class AdaptiveProfile(
    val longTermHrTrimBpm: Float = 0f,      // ← systematic HR bias (session-learned)
    val responseLagSec: Float = 25f,        // ← how fast your HR responds (default 25s)
    val paceHrBuckets: Map<Int, PaceHrBucket>, // ← pace→HR lookup table
    val totalSessions: Int = 0,            // ← confidence gating
    val ctl: Float = 0f,                    // ← chronic training load (fitness)
    val atl: Float = 0f,                    // ← acute training load (fatigue)
    val hrMax: Int? = null,                 // ← calibrated max HR
    val hrMaxIsCalibrated: Boolean = false,
    val hrRest: Float? = null,              // ← resting HR estimate
    val lastTuningDirection: TuningDirection?,
)

The three fields used by the trim system are highlighted in the first four lines: longTermHrTrimBpm, responseLagSec, paceHrBuckets, and totalSessions.

📡 Where the Adaptive Data Gets Used

The adaptive system feeds into three distinct output channels:

1. WorkoutState → UI

Every tick, the service writes the TickResult into WorkoutState (a singleton StateFlow):

WorkoutState.update {
  it.copy(
    predictedHr = adaptiveResult.predictedHr,
    guidanceText = guidance,
    adaptiveLagSec = adaptive.currentLagSec(),
    projectionReady = adaptiveResult.hasProjectionConfidence,
  )
}

The WorkoutScreen observes this StateFlow and renders:

  • The predicted HR value (when projection is ready)
  • The guidance text ("HR drifting up — ease off slightly")
  • The current response lag (for debugging/display)

📄 WorkoutForegroundService.kt:378–382

2. CoachingEventRouter → Audio Alerts

The CoachingEventRouter receives the TickResult and decides whether to fire audio coaching events:

  • Uses projected zone status (not just actual zone) to trigger preemptive alerts
  • Passes guidance text through to spoken/toned alerts
  • Alert timing influenced by the adaptive lag estimate

This is the key value: you hear "ease off" before your HR actually goes above zone, because the projection saw it coming.

📄 CoachingEventRouter.kt

3. Bootcamp ViewModels

BootcampViewModel and BootcampSettingsViewModel read the profile for:

  • hrMax — used to compute HR zones for bootcamp sessions
  • ctl/atl — fitness and fatigue signals for training load management
  • totalSessions — confidence indicator for the adaptive system

📄 BootcampViewModel.kt, BootcampSettingsViewModel.kt

4. FitnessSignalEvaluator

At workout end, the service runs a fitness evaluation pass using the updated profile:

  • Reads longTermHrTrimBpm, responseLagSec, ctl, atl
  • Produces a TuningDirection (harder/easier/maintain) stored back into profile
  • Used by bootcamp system to auto-adjust future session difficulty
